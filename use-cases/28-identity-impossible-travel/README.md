# UC-28: Normalized Identity Impossible Travel Detection

Straight Python demo showing why identity logs should be normalized before detection logic runs.

The point is simple:

```text
Entra ID + AD + Okta + VPN
        ↓
normalize to one schema
        ↓
run one impossible-travel detection
```

No Flink needed for this example. The demo focuses on the pipeline idea behind the LinkedIn post: normalize once, detect once.

---

## What This Shows

Most SOC teams collect identity logs from several sources, but each source uses different field names:

```text
Entra ID: userPrincipalName, createdDateTime, ipAddress
Okta: actor.alternateId, published, client.ipAddress
VPN: username, timestamp, remote_ip
AD: TargetUserName, TimeCreated, IpAddress
```

`normalize.py` maps those into one schema:

```text
event_time
event_type
source_system
user_id
src_ip
city
country
latitude
longitude
auth_result
mfa_result
device_trust
asn_reputation
risk_score
```

Then `detect-impossible-travel.py` runs one rule across all sources.

## Files

```text
raw-events/                    # Vendor-specific identity logs
normalize.py                   # Maps vendor fields to one schema
detect-impossible-travel.py    # Runs one detection over normalized events
normalized-events.json         # Generated by normalize.py
detections.json                # Generated by detect-impossible-travel.py
LINKEDIN_POST.md               # Short post copy for publishing
```

## Production Version

In production, this same idea becomes:

```text
Entra ID / AD / Okta / VPN
        ↓
OpenTelemetry Collector
        ↓
Normalize and enrich identity fields
        ↓
Kafka for real-time detection
        ↓
ClickHouse for investigation queries
        ↓
S3 for low-cost compliance archive
```

The important part is not the demo runtime. The important part is moving vendor-specific mapping out of every detection rule.
